## LAYER 5: ENGINE INTEGRITY

Audited all files with monetary operations. Revenue engine demonstrates strong Decimal.js discipline.

**Verdict: PASS** — 0 Blockers. All backend monetary arithmetic uses Decimal.js correctly.

### Warnings

1. **[WARNING] capacity-engine.ts: Native arithmetic on utilization percentage**
    - Code: `const utilization = (input.headcount / totalCapacity) * 100;`
    - Issue: Native float division for percentage (non-monetary); could produce imprecise display
    - File: `apps/api/src/services/capacity-engine.ts:64-65`
    - Fix: Use `new Decimal(input.headcount).div(totalCapacity).times(100).toDecimalPlaces(1)`

2. **[WARNING] capacity-engine.ts: Native \* with Decimal-sourced plafondPct**
    - Code: `Math.floor(totalCapacity * config.plafondPct) - input.headcount`
    - Issue: `plafondPct` originates from Prisma `Decimal(5,4)` but is coerced to `Number()` upstream
    - File: `apps/api/src/services/capacity-engine.ts:75`

3. **[WARNING] capacity-engine.ts: Math.round on utilization**
    - Code: `Math.round(utilization * 10) / 10`
    - Issue: Fragile rounding pattern (e.g., `Math.round(1.005 * 100) / 100` = 1, not 1.01)
    - File: `apps/api/src/services/capacity-engine.ts:83`

4. **[WARNING] enrollment/calculate.ts: Number() coercion on Prisma Decimal**
    - Code: `plafondPct: Number(gl.plafondPct)`
    - Issue: Converts Prisma Decimal to Number before passing to capacity engine
    - File: `apps/api/src/routes/enrollment/calculate.ts:60`
    - Fix: Pass as string, use Decimal in capacity engine

5. **[WARNING] revenue-engine.ts + other-revenue.ts: weightArray as number[]**
    - Code: `weightArray?: number[] | null` / `z.array(z.number()).length(12)`
    - Issue: Distribution weights accepted as IEEE 754 floats; mitigated by last-bucket pattern
    - File: `apps/api/src/services/revenue-engine.ts:42`, `apps/api/src/routes/revenue/other-revenue.ts:40`
    - Fix: Accept weights as `z.array(z.string().regex(/^\d+(\.\d+)?$/))` for full precision

6. **[WARNING] versions.ts: Monetary strings use .toString() instead of .toFixed(4)**
    - Code: `total_revenue_ht: pRevTotal.toString()`
    - Issue: Produces variable decimal places ("1234.89") vs standardized ("1234.8900")
    - File: `apps/api/src/routes/versions.ts:298-324`
    - Fix: Use `.toFixed(4)` for consistency with all other API monetary responses

7. **[WARNING] other-revenue-tab.tsx: Frontend summation uses native arithmetic**
    - Code: `items.reduce((sum, i) => sum + Number(i.annualAmount), 0).toFixed(4)`
    - Issue: Per CLAUDE.md, frontend should use `decimal.js` for display rounding, never native calc
    - File: `apps/web/src/components/revenue/other-revenue-tab.tsx:149`
    - Fix: `items.reduce((acc, i) => acc.plus(new Decimal(i.annualAmount)), new Decimal(0)).toFixed(4)`

## LAYER 6: CROSS-EPIC CONSISTENCY

Checked coherence across all 7 completed epics. Overall strong architectural alignment.

### Warnings

1. **[WARNING] API error response format inconsistency**
    - Auth routes return `{ error: 'INVALID_CREDENTIALS', message: '...' }`
    - Data routes return `{ code: 'VERSION_LOCKED', message: '...' }`
    - Frontend `apiClient` checks `body.code` but auth uses `error` field
    - Files: `apps/api/src/routes/auth.ts:63`, `apps/web/src/lib/api-client.ts:24`
    - Fix: Standardize all error responses to `{ code: string, message: string }` format

2. **[WARNING] Frontend mutation errors not shown to users**
    - `useMutation` hooks in `use-enrollment.ts` and `use-revenue.ts` have no `onError` callback
    - If a PUT fails (e.g., 422 VAT_MISMATCH), user sees no visual feedback
    - Files: `apps/web/src/hooks/use-enrollment.ts:66-78`, `apps/web/src/hooks/use-revenue.ts:24-36`
    - Fix: Add `onError` callback to all mutations that displays toast notification

3. **[WARNING] Logout fetch error silently swallowed**
    - `auth-store.logout()` uses `fetch()` directly (not `apiClient`) and catches + ignores errors
    - Other API calls use `apiClient` with proper error handling
    - File: `apps/web/src/stores/auth-store.ts:96-102`
    - Fix: Use `apiClient` for consistency, or add explicit comment explaining why failure is ignored

### Info

4. **[INFO] No React ErrorBoundary wrapping page components**
    - If a page component throws during render, entire app crashes with white screen
    - File: `apps/web/src/App.tsx`
    - Fix: Add `<ErrorBoundary>` wrapper around `<RouterProvider>` or per-route

5. **[INFO] Historical enrollment route is global (not version-scoped) — correct by design**
    - `/api/v1/enrollment/historical` is not under `/versions/:versionId/` prefix
    - This is intentional: historical data is master data, not version-specific

6. **[INFO] Auth store login/refresh use fetch() directly — intentional**
    - These endpoints cannot use apiClient since no access token exists yet

7. **[INFO] Tariff type restriction: DiscountEntry excludes 'Plein' — correct by design**
    - FeeGridEntry allows `['RP', 'R3+', 'Plein']` but DiscountEntry only `['RP', 'R3+']`
    - Plein tariff (full fee) intentionally receives no discounts
