# Feature Spec: Data Migration

> Epic: #10 | Phase: 4 — SPECIFY | Status: Draft | Date: 2026-03-09

## Summary

Epic 12 delivers the one-time data migration pipeline that imports EFIR's historical and planning data into the BudFin database. TypeScript migration modules within the existing monorepo (`apps/api/src/migration/`) read from pre-parsed JSON fixtures (generated by existing TypeScript parsers) and historical enrollment CSVs, then insert data via Prisma ORM operations. All monetary arithmetic uses `Decimal` from `decimal.js` (mirroring TC-001 at the migration layer). The pipeline includes a pre-migration `pg_dump` backup, idempotent script execution, and a 6-step automated validation protocol enforcing zero-tolerance data integrity against source files. Successful migration is a go-live gate: revenue totals must match within +/- 1 SAR, staff cost totals within +/- 1 SAR, and enrollment headcounts must be an exact match. Migration targets the 4 core data domains already implemented (Enrollment, Revenue, DHG, Staff Costs); P&L migration is deferred until Epic 5 is complete. The only new parser required is for the Staff Costs XLSX workbook — all other source data already exists as JSON fixtures.

> **Architecture Decision**: This spec uses TypeScript migration scripts within the monorepo (`apps/api/src/migration/`) instead of standalone Python + psycopg2 scripts. Rationale:
>
> 1. JSON fixtures already exist — Excel parsing was completed by existing TypeScript parsers
> 2. `seed-fy2026.ts` is a working reference implementation of the exact import pattern
> 3. Prisma client handles all DB operations; raw SQL only needed for pgcrypto encryption
> 4. `decimal.js` provides TC-001 compliant fixed-point arithmetic (equivalent to Python `decimal.Decimal`)
> 5. Keeps migration in the same language/toolchain as the rest of the codebase
> 6. Only exception: Staff Costs XLSX requires a new TypeScript parser (Story 2) since no fixture exists

## Acceptance Criteria

Each criterion must be testable and specific. Use "Given / When / Then" or plain language.

- [ ] AC-01: Given source JSON fixtures and CSVs, when migration scripts execute, then all monetary values are read using `new Decimal(stringValue)` from `decimal.js` and stored as `DECIMAL(15,4)` in PostgreSQL. No raw `number` types are used for monetary values in any migration script. No `parseFloat()` for monetary values.
- [ ] AC-02: Given the fee grid fixture (`fy2026-fee-grid.json`), when `importers/revenue.ts` completes, then the sum of all `fee_grids.tuitionHt` values per grade per period matches the source within +/- 1 SAR. Fee grid HT = TTC / (1 + VAT_RATE) is validated for every imported row.
- [ ] AC-03: Given the staff costs fixture (`fy2026-staff-costs.json`), when `importers/employees.ts` completes, then the sum of all employee base salary + allowances matches the source within +/- 1 SAR. All 6 salary/compensation fields (`baseSalary`, `housingAllowance`, `transportAllowance`, `responsibilityPremium`, `hsaAmount`, `augmentation`) are encrypted via `pgp_sym_encrypt()` using Prisma `$executeRawUnsafe` and stored as `Bytes`.
- [ ] AC-04: Given 5 historical enrollment CSVs, when `importers/enrollment-actuals.ts` completes, then enrollment headcount per grade per academic period is an exact integer match against the CSV source (zero tolerance).
- [ ] AC-05: Given the Staff Costs XLSX (the only unparsed source), when the parser encounters dirty data (formula errors `#REF!`/`#DIV/0!`/`#N/A`, empty mandatory fields, invalid types), then the script logs the issue as a structured JSON warning, skips or coerces the value per documented rules, and the final validation report flags all data quality exceptions. All other source data (JSON fixtures and CSVs) is pre-validated and clean.
- [ ] AC-06: Given a target PostgreSQL database, when the migration runner starts, then a `pg_dump` backup is created before any data modification. The backup is restorable to the pre-migration state within 5 minutes.
- [ ] AC-07: Given a migration script that has already run, when the same script is re-executed, then it produces the same end state (idempotent). Existing rows are upserted, not duplicated. Row counts remain unchanged.
- [ ] AC-08: Given 5 enrollment CSVs, when `importers/enrollment-actuals.ts` runs, then it creates one `budget_versions` row per academic year with `type='Actual'`, `dataSource='IMPORTED'`, `status='Locked'`, and populates `enrollment_headcount` rows under each version. Historical CSVs use `academicPeriod='AY1'` for annual totals (CSVs are not period-split).
- [ ] AC-09: Given the DHG fixture (`fy2026-dhg-structure.json`), when `importers/dhg-grille.ts` completes, then all `dhg_grille_config` rows match the source: `gradeLevel`, `subject`, `hoursPerWeekPerSection`, and `dhgType` are correct. Subject names are preserved as-is from the fixture (including prefixes like "[Host-Country]").
- [ ] AC-10: Given the other revenue fixture (`fy2026-other-revenue.json`), when `importers/other-revenue.ts` completes, then all `other_revenue_items` rows match the source: `lineItemName`, `annualAmount`, `distributionMethod`, and `ifrsCategory` are correct. Both positive and negative amounts (scholarship deductions) are imported.
- [ ] AC-11: Given all migration importers have completed, when the 6-step validation protocol runs, then all 6 checks pass: (1) row count match, (2) column checksum SHA-256, (3) revenue totals +/- 1 SAR, (4) staff cost totals +/- 1 SAR, (5) P&L reconciliation skipped (deferred), (6) enrollment cross-check exact match.
- [ ] AC-12: Given migration completion, when the `actuals_import_log` table is queried, then one audit row exists per migration module execution with `sourceFile`, `rowsImported`, `validationStatus`, and `importedAt` populated.
- [ ] AC-13: Given all migration modules, when any module executes, then structured JSON logs are written with: module name, start/end timestamps, row counts per table, warnings, errors, and duration. Logs are sufficient for post-migration audit review.
- [ ] AC-14: Given a fresh database, when the migration scaffold initializes, then a system user (e.g., `migration@system`) exists in the `users` table via Prisma `upsert` to satisfy `createdBy`, `updatedBy`, and `importedById` foreign key constraints on all migrated rows and `actuals_import_log` entries.

## Known Risks

1. **Staff Costs XLSX structure unknown** — Story 2 must start with ExcelJS sheet discovery before coding the parser. The `import.ts` column mappings guide expected fields, but Excel headers may differ.
2. **Term amounts not in fee grid fixture** — `term1Amount`, `term2Amount`, `term3Amount` default to 0. These can be computed post-migration by the revenue engine.
3. **DHG nested expansion complexity** — The `fy2026-dhg-structure.json` has 4 different nesting patterns (maternelle hours, elementaire hours, college flat, lycee per-level). Each needs a separate expansion loop.
4. **Decimal precision mix** — Fixtures use 2dp, 4dp, and 6dp. All migration code must use `new Decimal(string)` not `parseFloat()` for monetary values.
5. **Other revenue has negative amounts** — 2 Bourses items are negative (scholarship deductions). The Prisma model `annualAmount DECIMAL(15,4)` accepts negatives — no issue, but validation must not flag these as errors.

## Out of Scope

- **P&L migration data**: Epic 5 (P&L Reporting) is not yet implemented. The `monthly_budget_summary` table will be populated by P&L calculation engines after Epic 5, not by migration. Validation step 5 (P&L reconciliation) is skipped.
- **Ongoing actuals import**: The two-phase import API (`/api/v1/versions/:id/import/:module/validate` and `/commit`) is for post-go-live actuals imports, not for one-time migration. Migration scripts use Prisma directly, not the API.
- **UI for migration**: No admin UI to trigger or monitor migration. Scripts run via CLI only.
- **Real-time sync with Excel**: Migration is a one-time operation. No ongoing sync between Excel workbooks and BudFin database.
- **Historical calculated data**: Migration imports raw input data only (headcounts, fees, employee records, DHG hours). Calculated outputs (monthly revenue, monthly staff costs, EoS provisions, DHG requirements) are recomputed by BudFin's engines after import.
- **Scenario data migration**: No scenario versions are imported. Only one Budget version (FY2026) and 5 Actual versions (historical AYs) are created.
- **Data archival or purge**: No cleanup of source Excel/CSV files after migration.
